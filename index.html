CASE 01 / 04

3PL 전환 이슈 대응 및 재배송·취소 1,928건 우선순위·일정 관리

우선순위 설계일정·리스크 관리리소스 재산정
내 역할 · 전체 처리 대상 분류, 고객 영향도 기준 우선순위 설계, 자동화 실행, 잔여 작업량·인력 재산정, 유관부서 조율

Situation

금요일 3PL 전환 과정에서 대량 재배송·취소 후속 처리가 발생해 총 1,928건의 처리 대상과 완료 일정을 다시 정리해야 했습니다. 이 중 재배송 대상 1,151건은 이미 한 차례 배송 불편을 겪은 고객에게 다시 안내한 도착 약속일이 있었고, 주말 동안 일반 주문이 늘면 고객에게 약속한 도착일의 주문 가능 물량이 마감될 위험이 있었습니다. 취소 대상 777건은 월요일에 처리해도 고객 영향이 상대적으로 작은 상황이었습니다.

Success Gap

접수 순서나 전체 물량 기준으로만 작업을 배분하면 고객 약속 일정이 걸린 재배송 건이 상대적으로 늦게 처리될 위험이 있었습니다. 처음에는 전체 물량을 기준으로 지원 인력 3명을 요청한 상태였지만, 재배송과 취소의 긴급도 차이를 반영해 작업 순서와 필요 인력을 다시 설계할 필요가 있었습니다.

What I Tracked

재배송·취소 건별 고객 영향도와 약속일 마감 위험 시점, 재배송 자동화 진행 상황, 남은 취소 물량과 처리 소요 시간, 필요 지원 인력 규모를 지속적으로 확인했습니다.

How I Coordinated

1
고객 영향도 기준으로 우선순위 재설계
처리량이 아니라 고객 영향도와 일정 리스크를 기준으로 재배송 1,151건과 취소 777건을 분리하고, 약속일 마감 위험이 있는 재배송을 최우선 과제로 지정했습니다.
2
병목 구간에 자동화를 적용해 핵심 작업 선행 완료
일정 지연의 병목 구간이었던 재배송 처리에 Chrome Extension과 Apps Script를 연결해 반복 입력 구간을 자동화하고, 핵심 작업을 주말 안에 선행 완료했습니다.
3
잔여 작업량과 인력을 다시 산정
재배송 자동화 완료 이후 남은 작업량이 취소 777건으로 줄어든 상황을 반영해 필요 지원 인력을 3명에서 2명으로 재산정하고, 월요일 오전 내 후속 처리를 마무리했습니다.

Outcome

Chrome Extension과 Apps Script를 활용해 재배송 1,151건을 주말 내 완료함으로써, 이미 불편을 겪은 고객에게 약속한 도착일의 주문 가능 물량이 마감되는 리스크를 먼저 차단했습니다. 이후 취소 대상 777건의 작업량을 다시 계산해 지원 인력을 기존 3명에서 2명으로 조정했고, 월요일 오전 전체 후속 처리를 마무리했습니다. 추가 인력에만 의존하지 않고 우선순위와 잔여 작업량에 따라 일정·리소스를 재조정한 경험입니다.

메디컬에이아이에서의 적용 · 이슈 대응 시에도 처리량이 아니라 고객 영향도와 일정 리스크를 기준으로 우선순위를 설계하고, 진행 상황에 따라 잔여 작업과 필요 리소스를 다시 추정해 조정하겠습니다.

CASE 02 / 04

사용자 문제를 기능 요구사항으로 전환한 제품 개선

요구사항 정의최소 구현안 설계개발팀 협의
내 역할 · 이상 문의 패턴 포착, 화면 구간 식별, 문제 재정의 및 요구사항 작성, 최소 구현안 설계, CTO·개발 담당자 협의

Situation

화요일에만 도착 가능한 신규 상품을 등록한 직후, 기존에는 없던 동일 유형의 문의가 짧은 시간에 집중됐습니다. 배송 가능 요일이 다른 상품을 함께 담으면 주문이 차단됐지만, 고객 화면에는 일반적인 품절형 메시지가 노출돼 어떤 상품 때문에 주문이 불가능한지 알기 어려운 상태였습니다.

Success Gap

실제 품절과 배송일 제약이 같은 방식으로 보이면서 고객은 문제를 스스로 해결할 수 없었고, 상담 담당자는 매번 상품과 배송 가능일을 다시 확인해 설명해야 했습니다. 주문 차단 로직 전체를 변경하지 않고도 고객이 원인을 이해할 수 있는 요구사항 정의가 필요했습니다.

What I Tracked

신규 상품 등록 시점과 문의 발생 시점, 문의 고객의 장바구니 구성, 실제 재고 여부, 화면에 노출된 메시지를 비교했습니다. 문의 건마다 같은 배송 제약 상품이 포함돼 있었고 실제 품절이 아니었기 때문에, 개별 주문 문제가 아니라 화면 안내 구조에서 발생한 공통 문제로 판단했습니다.

How I Coordinated

1
신규 상품 등록 직후 짧은 시간에 집중된 동일 유형 문의를 포착하고, 주문별 상품 구성과 노출 메시지를 비교해 공통 원인을 확인한 뒤 개발팀에 즉시 공유
2
주문 차단 로직 전체를 변경하기보다, 주의상품 태그와 상품별 배송 가능 요일을 장바구니에 노출하는 최소 구현안을 요구사항으로 작성하고 예시 문구 제안
3
CTO·개발 담당자와 구현 범위를 확인하고 당일 반영
4
반영 이후 관련 문의 추이를 확인하고, 유사 상품과 프로모션 안내에도 같은 구조를 확대 적용
실제 화면
장바구니 상품별 배송 제약 안내 subtext 반영 화면 요구사항대로 구현됐는지 확인하고, 어떤 상품 때문에 배송일이 맞지 않는지 장바구니 단계에서 바로 확인할 수 있도록 개선했습니다. 내부 정보 비식별 처리

Outcome

문제 확인 직후 상품별 배송 가능 요일을 장바구니에 안내하는 요구사항을 정리하고, CTO·개발 담당자와 협의해 당일 반영했습니다. 등록 당일 6~10건 발생한 관련 문의는 반영 이후 0건 수준으로 감소했으며, 동일한 안내 방식은 유사한 배송 제약 상품에도 확대 적용됐습니다.

메디컬에이아이에서의 적용 · 사용자가 기능을 사용하지 못하는 이유를 화면과 실제 사용 흐름에서 확인하고, 요구사항과 최소 구현 범위로 정리해 개발 조직과 빠르게 검증하겠습니다.

CASE 03 / 04

내부 운영 Workflow·자동화 도구 구축 및 QA

요구사항·예외조건 정의도구 구성·구축테스트·문서화
내 역할 · 자동화 도구 요구사항 정의, 예외조건 설계, Claude를 활용한 도구 구성·구축, 정상·오류·입력 예외 테스트, 사용 가이드 문서화

Situation

매장·타부서의 자유 형식 Slack 요청, 반복적인 재배송·반품 등록, 담당자·상품코드 조회, 환불 계좌 검증까지 여러 반복 업무가 수작업으로 운영되고 있었습니다. 각 업무마다 필요한 입력 항목과 예외 상황이 달라 자동화 전에 요구사항과 예외조건을 먼저 정리할 필요가 있었습니다.

Success Gap

자동화 도구를 만들더라도 예외조건과 테스트 항목이 명확하지 않으면 오류 발생 시 원인 파악이 어렵고, 담당자가 바뀌면 사용 방법이 제대로 전달되지 않을 위험이 있었습니다.

What I Tracked

업무별 입력 항목, 처리 예외 상황(귀책·비용 판단이 필요한 구간), 정상·오류 케이스 테스트 결과, 실제 적용 이후 사용자 피드백을 확인했습니다.

How I Coordinated

1
요청 접수 요구사항 표준화
Slack Workflow Builder로 작성자·담당자·상품정보·문의·요청·이미지 항목을 정의하고 표준 접수 양식을 구성했습니다. 접수 내용은 담당자를 태그해 지정 채널에 게시하고, Slack Workflow의 네이티브 Google Sheets 연동 기능으로 시트에 자동 기록되도록 연결했습니다.
2
반복 처리 구간 자동화
Claude를 활용해 Chrome Extension과 Apps Script 기반 자동화 도구를 구성하고, 재배송 등록은 6→2단계, 반품 등록은 7→1단계로 단축했습니다. 귀책·비용 판단이 필요한 구간은 예외 기준으로 남겨 담당자가 직접 확인하도록 설계했습니다.
3
조회·검증용 API 연동 도구 구축
Slack API 기반 담당자·상품코드 조회봇과 Claude를 활용한 Python 기반 외부 예금주 조회 API 연동 도구를 구축하고, 정상 응답·오류 응답·입력값 누락 등 예외 케이스를 테스트했습니다.
4
요구사항·테스트 결과 문서화
도구별 요구사항, 입력값, 예외조건, 테스트 결과, 사용 절차를 문서화해 실제 운영에 적용하고, 사용 과정에서 확인된 오류와 불편 사항을 반영했습니다.
실제 화면

사용자 피드백 173건 분석 및 제품·운영 대안 설계

피드백 통합 분석AS-IS/TO-BE 설계고객 구제 기준 수립
내 역할 · 12월 문의 증가 패턴 확인, 이전 행사 피드백 통합, 173건 분류·분석, AS-IS/TO-BE 개선안 작성, 구현 제약 검토, 고객 구제 기준 설계

Situation

1+1·2+1 프로모션이 진행될 때마다 일부 고객은 증정 수량까지 직접 장바구니에 담아야 한다는 사실을 알지 못한 채, 상품 1개만 담아도 혜택이 자동으로 적용된다고 인식했습니다. 조건을 충족하지 않은 상태에서는 부족 수량이나 필요한 행동이 안내되지 않아 고객이 결제 전에 오인을 알아차리기 어려웠고, 이미 혜택을 받았다고 생각한 상태에서 문의해 클레임 강도도 높은 편이었습니다. 12월 행사에서 같은 유형의 문의가 다시 증가하면서 반복 원인을 구조적으로 확인할 필요가 생겼습니다.

Success Gap

이전에도 관련 피드백이 전달됐지만 행사 빈도, 전체 판매량 대비 문의 비중, 다양한 수량 조합과 UI 충돌 가능성 등의 이유로 추가 개선 우선순위를 확보하기 어려웠습니다. 단순히 문의가 많다는 설명을 반복하기보다, 이전 행사 이력까지 같은 기준으로 분석해 고객이 실제로 어디에서 조건을 오인하는지 보여주고 제품 개선과 운영 대안 중 현실적인 해결 방식을 결정할 필요가 있었습니다.

What I Tracked

12월 행사 문의와 이전 프로모션 기간의 상담 이력을 함께 모아 피드백 173건을 인입 원인, 발생 채널, 고객 행동 시점, 정책 인지 여부로 재분류했습니다. 분석 결과 143건(82%)이 행사 조건과 수량 입력 방식 미인지에서 발생했고, 94.8%가 모바일 채널에 집중돼 있음을 확인했습니다. 이를 통해 개별 고객의 단순 실수가 아니라 같은 화면 구조에서 반복되는 오인임을 확인했습니다.

How I Coordinated

1
12월 문의 증가를 계기로 해당 기간의 피드백만 보지 않고 이전 행사 상담 이력까지 173건을 같은 기준으로 통합 분류
2
조건 미충족 상태에서는 필요한 행동이 보이지 않는 문제를 정리하고, "혜택을 받으려면 X개를 더 담아주세요"와 같이 부족 수량을 안내하는 AS-IS/TO-BE 개선안 작성
3
EC 리더와 PC·모바일 구조 차이, 수량 조합별 예외 조건, UI 충돌 가능성, 행사 빈도와 판매량 대비 피드백 비중을 함께 검토해 즉시 구현 여부 판단
4
추가 UI 변경이 바로 진행되기 어려운 상황에서 초회 인입 고객에게는 누락 증정품을 추가 발송하고, 동일 사유 재문의부터는 행사 조건을 안내한 뒤 종결하는 고객 구제 기준 수립

Outcome

추가 UI 구현이 즉시 어렵다는 제약을 확인한 뒤, 초회 고객에게는 누락 증정품을 추가 발송하고 동일 사유 재문의부터는 행사 조건을 안내하는 기준을 수립했습니다. 건당 3천 원 미만의 추가 배송비는 발생했지만, 이미 혜택을 받았다고 인식한 고객의 고강도 클레임이 장기화되는 것보다 빠르게 불만을 해소하는 편이 효율적이라고 판단했습니다. 이를 통해 상담원의 반복 설명과 감정 소모를 줄이면서 제품 제약 안에서 고객 경험을 보호했습니다.
